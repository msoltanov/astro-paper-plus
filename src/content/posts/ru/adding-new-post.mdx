## Создание записи в блоге

Чтобы написать новую запись, создайте файл Markdown (или MDX) в каталоге `src/content/posts/`. Записи можно группировать в подкаталоги для удобства. Имя подкаталога становится частью URL записи. Например, `src/content/posts/2025/example-post.md` будет доступен по адресу `/posts/2025/example-post`. Если нужен подкаталог только для организации и он не должен влиять на URL, добавьте к его имени префикс-подчёркивание (`_`).

```bash
# Пример: пути к файлам записей и соответствующие URL
src/content/posts/very-first-post.md -> mysite.com/posts/very-first-post
src/content/posts/2025/example-post.md -> mysite.com/posts/2025/example-post
src/content/posts/_2026/another-post.md -> mysite.com/posts/another-post
src/content/posts/docs/_legacy/how-to.md -> mysite.com/posts/docs/how-to
src/content/posts/Example Dir/Dummy Post.md -> mysite.com/posts/example-dir/dummy-post

```

> [!TIP]
> Файлы и каталоги с префиксом `_` исключаются из маршрутизации. Используйте их для черновиков, общих ассетов и внутренних материалов.

## Frontmatter

Frontmatter — основное место для метаданных записи в блоге. Он располагается в начале файла в формате YAML. Подробнее о frontmatter и его использовании читайте в [документации Astro](https://docs.astro.build/en/guides/markdown-content/). Вот список свойств frontmatter для каждой записи:

<ResponsiveTable variant="striped-minimal">
| Свойство              | Описание                                                                                                                                       | Примечание                                       |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------ |
| **_title_**           | Заголовок записи. (h1)                                                                                                                         | обязательно<sup>\*</sup>                        |
| **_description_**     | Описание записи. Используется в анонсе и мета-тегах описания страницы. Если поле опущено, AstroPaper+ возьмёт анонс из тела записи до разделителя `<!-- more -->`. | необязательно (рекомендуется для SEO)            |
| **_pubDatetime_**     | Дата и время публикации в формате ISO 8601.                                                                                                    | обязательно<sup>\*</sup>                        |
| **_modDatetime_**     | Дата и время последнего изменения в формате ISO 8601. (добавляйте это свойство только когда запись была изменена)                                | необязательно                                   |
| **_author_**          | Автор записи.                                                                                                                                  | по умолчанию = `site.author`                    |
| **_featured_**        | Показывать ли запись в блоке «Избранное» на главной странице.                                                                                  | по умолчанию = false                            |
| **_draft_**           | Пометить запись как «неопубликованную».                                                                                                         | по умолчанию = false                            |
| **_tags_**            | Связанные ключевые слова для записи. Записываются в виде массива YAML.                                                                         | по умолчанию = others                           |
| **_ogImage_**         | OG-изображение записи. Полезно для шеринга в соцсетях и SEO. Может быть удалённым URL или путём к изображению относительно текущей папки.     | по умолчанию = `site.ogImage` или сгенерированное OG-изображение |
| **_canonicalURL_**    | Канонический URL (абсолютный), если статья уже опубликована в другом источнике.                                                                | по умолчанию = `Astro.site` + `Astro.url.pathname` |
| **_hideEditPost_**    | Скрывать кнопку редактирования под заголовком записи. Применяется только к текущей записи.                                                     | по умолчанию = false                            |
| **_timezone_**        | Указать часовой пояс в формате IANA для текущей записи. Перекрывает глобальный `site.timezone` только для этой записи.                          | по умолчанию = `site.timezone`                  |
</ResponsiveTable>

> [!TIP]
> Получить дату и время в формате ISO 8601 можно, выполнив `new Date().toISOString()` в консоли.

Обязательными являются только поля `title` и `pubDatetime`. Поле `description` — необязательное: если его нет, AstroPaper+ возьмёт анонс из тела записи до разделителя `<!-- more -->` (тот же синтаксис, что в Jekyll/Hugo/Eleventy). Заголовок и описание (анонс) по-прежнему важны для поисковой оптимизации (SEO), поэтому AstroPaper+ рекомендует включать в каждую запись либо `description` во frontmatter, либо маркер `<!-- more -->` в теле. Если опустить `tags` в записи (то есть не указать ни одного тега), будет использован тег по умолчанию — `others`. Тег по умолчанию можно задать в `src/content.config.ts`:

```ts file="src/content.config.ts"
// ...
tags: z.array(z.string()).default(["others"]), // замените "others" на нужное значение
// ...

```

### Пример frontmatter

Вот пример frontmatter для записи:

```yaml file="src/content/posts/sample-post.md"
---
title: Заголовок записи
author: ваше имя
pubDatetime: "2022-09-21T05:17:19Z"
featured: true
draft: false
tags:
  - какой-то
  - пример
  - тегов
ogImage: ../../assets/images/example.png # src/assets/images/example.png
# ogImage: "https://example.org/remote-image.png" # удалённый URL
description: Это пример описания для примера записи.
canonicalURL: https://example.org/my-article-was-already-posted-here
---

```

### Сниппеты VS Code (необязательно)

AstroPaper+ включает сниппеты для рабочего пространства, ускоряющие создание новых записей:

- **frontmatter**: вставляет рекомендуемый блок frontmatter
- **template**: вставляет базовый шаблон записи (включая `## Table of contents`)

Эти сниппеты лежат в `.vscode/astro-paper.code-snippets`. Если вы используете VS Code (или Cursor), они должны быть доступны автоматически при открытии рабочего пространства.

## Блок-выноски (callouts)

AstroPaper+ поддерживает блок-выноски, начиная с upstream AstroPaper+ v6.1; они присутствуют во всех релизах AstroPaper+ от v6.1.0 и выше (включая **AstroPaper+ v7.0.0**, релиз этого форка). Они используют простой синтаксис blockquote на базе `rehype-callouts` (тема Obsidian). Вот наиболее часто используемые типы:

> [!NOTE]
> Дополнительная информация, о которой читатель должен знать.

> [!TIP]
> Полезный совет, сокращение или лучшая практика.

> [!WARNING]
> Что-то, что может пойти не так или иметь непредвиденные последствия.

> [!DANGER]
> Серьёзный риск сбоя, потери данных или некорректного поведения.

> [!INFO]
> Нейтральный информационный контекст — менее срочный, чем note.

> [!SUCCESS]
> Подтверждение того, что что-то сработало или корректно.

Полный список поддерживаемых типов: `NOTE`, `ABSTRACT`, `INFO`, `TODO`, `TIP`, `SUCCESS`, `QUESTION`, `WARNING`, `FAILURE`, `DANGER`, `BUG`, `EXAMPLE`, `QUOTE` — у каждого своя иконка и цвет. Многие типы также принимают псевдонимы (например, `HINT` и `IMPORTANT` для `TIP`, `CAUTION` для `WARNING`). См. [документацию rehype-callouts](https://github.com/lin-stephanie/rehype-callouts) для полного справочника.

### Сворачиваемые блок-выноски

Добавьте `-` после типа, чтобы выноска была свёрнута по умолчанию, или `+`, чтобы она была раскрыта, но сворачиваемой:

> [!WARNING]- Прочтите перед продолжением
> Это содержимое скрыто, пока читатель не раскроет выноску. Удобно для длинных оговорок, которые иначе прерывали бы поток.

> [!TIP]+ Совет (раскрыт по умолчанию)
> Этот блок открыт изначально, но может быть свёрнут. Подходит для необязательных деталей, которые вы всё же хотите показать при первой загрузке.

### Пользовательские заголовки

Замените метку типа по умолчанию на любой нужный заголовок, добавив текст после типа:

> [!NOTE] А вы знали?
> Текст после типа становится заголовком выноски. Если его не указать, используется имя типа автоматически.

### Сводка по синтаксису

```md
> [!NOTE]
> Дополнительная информация.

> [!WARNING]- Свёрнуто по умолчанию
> Скрыто до раскрытия.

> [!TIP]+ Раскрыто, но сворачиваемо
> Открыто изначально.

> [!DANGER] Пользовательский заголовок
> Заменяет стандартный заголовок.

```

## Добавление оглавления

По умолчанию запись не содержит оглавления (Table of contents — TOC). Чтобы добавить его, напишите `Table of contents` как заголовок h2 (`##` в Markdown) и поместите туда, где хотите его видеть:

```md
---
# frontmatter
---

Здесь приведены рекомендации, советы и приёмы по созданию новых записей в теме AstroPaper+. <!-- [!code ++] -->

## Table of contents

<!-- остальная часть записи -->

```

## Заголовки

Стоит отметить один момент насчёт заголовков. Записи блога AstroPaper+ используют `title` (из frontmatter) как главный заголовок записи. Поэтому остальные заголовки в записи должны использовать `h2` \~ `h6`. Это правило необязательно, но настоятельно рекомендуется по соображениям визуальной согласованности, доступности и SEO.

## Подсветка синтаксиса

AstroPaper+ использует [Shiki](https://shiki.style/) в качестве стандартного подсветчика синтаксиса вместе с [@shikijs/transformers](https://shiki.style/packages/transformers) для расширенных возможностей fenced-блоков кода. Если вы не хотите использовать эти трансформеры, их можно удалить:

```bash
pnpm remove @shikijs/transformers

```

```ts file="astro.config.ts"
// ...
// [!code --:5]
import { transformerNotationDiff, transformerNotationHighlight, transformerNotationWordHighlight,
} from "@shikijs/transformers";

export default defineConfig({
  // ...
  markdown: {
    remarkPlugins: [remarkToc, [remarkCollapse, { test: "Table of contents" }]],
    shikiConfig: {
      themes: { light: "min-light", dark: "night-owl" },
      defaultColor: false,
      wrap: false,
      transformers: [
        transformerFileName(), // [!code --:3]
        transformerNotationHighlight(),
        transformerNotationWordHighlight(),
        transformerNotationDiff({ matchAlgorithm: "v3" }),
      ],
    },
  },
  // ...
});

```

## Хранение изображений для контента блога

Есть два способа хранить изображения и использовать их в файле Markdown.

> [!IMPORTANT]
> Если вам нужно стилизовать оптимизированные изображения в Markdown, используйте [MDX](https://docs.astro.build/en/guides/images/#images-in-mdx-files).

### В каталоге `src/assets/` (рекомендуется)

Изображения можно хранить в каталоге `src/assets/`. Эти изображения будут автоматически оптимизированы Astro через [Image Service API](https://docs.astro.build/en/reference/image-service-reference/). Для ссылки на них можно использовать относительный путь или алиас (`@/assets/`). Пример: допустим, вы хотите показать `example.jpg`, который лежит по пути `src/assets/images/example.jpg`.

```md
![что-то](@/assets/images/example.jpg)
<!-- ИЛИ -->
![что-то](../../assets/images/example.jpg)
<!-- Тег img или компонент Image в Markdown не сработают ❌ -->
<img src="@/assets/images/example.jpg" alt="что-то">
<!-- ^^ Это неправильно -->

```

> [!TIP]
> Технически изображения можно хранить в любом каталоге под `src`. `src/assets` — лишь рекомендация.

### В каталоге `public/`

Изображения можно хранить в каталоге `public/`. Имейте в виду, что изображения в `public/` остаются без изменений со стороны Astro — то есть они не оптимизируются, и оптимизацию нужно делать вручную. Для таких изображений используйте абсолютный путь. Их можно отображать с помощью [синтаксиса изображений Markdown](https://www.markdownguide.org/basic-syntax/#images-1) или HTML-тега `img`. Пример: допустим, `example.jpg` расположен по пути `public/assets/images/example.jpg`.

```md
![что-то](/assets/images/example.jpg)
<!-- ИЛИ -->
<img src="/assets/images/example.jpg" alt="что-то">

```

## Бонус

### Сжатие изображений

> [!WARNING]
> Перед размещением изображений в записи блога (особенно в каталоге `public/`) сжимайте их. Неоптимизированные изображения существенно ухудшают производительность страниц. Рекомендуемые сервисы сжатия изображений:
>
> - [TinyPng](https://tinypng.com/)
> - [TinyJPG](https://tinyjpg.com/)

### OG-изображение

Если в записи не указано OG-изображение, используется OG-изображение по умолчанию. Хотя это необязательно, для записи следует указать релевантное OG-изображение в frontmatter. Рекомендуемый размер OG-изображений — **_1200 × 640_** px.

> [!TIP]
> Начиная с AstroPaper+ v1.4.0, OG-изображения генерируются автоматически, если они не указаны. Загляните в анонс.
